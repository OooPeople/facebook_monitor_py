# 使用說明

這份文件提供較完整的安裝、設定與操作說明。

如果你只是第一次快速了解專案，請先看：

- [`../README.md`](../README.md)

## 安裝

### 1. 安裝 Tampermonkey

先在瀏覽器安裝 [Tampermonkey](https://chromewebstore.google.com/detail/dhdgffkkebhmkfjojejmpbldmpobfkfo?utm_source=item-share-cb)。

建議把它釘選在瀏覽器右上角，之後比較容易確認腳本是否啟用。

### 2. 安裝腳本

1. 從瀏覽器右上角開啟 Tampermonkey
2. 進入 `控制台`
3. 建立新的腳本
4. 將 [`../src/facebook_group_refresh.user.js`](../src/facebook_group_refresh.user.js) 內容完整貼上
5. 儲存腳本

### 3. 開啟 Facebook 社團頁

此腳本只會在下列網址格式啟用：

```text
https://www.facebook.com/groups/<group-id>/
https://www.facebook.com/groups/<group-id>/posts/<post-id>
https://www.facebook.com/groups/<group-id>/permalink/<post-id>
```

進入社團頁後，建議手動重新整理一次，讓腳本完整初始化並顯示右上角面板。

如果重新整理後仍沒看到面板，先檢查：

- Tampermonkey 是否啟用這支腳本
- 目前頁面是否真的是 `www.facebook.com/groups/*`

## 主面板

腳本啟動後，頁面右上角會出現控制面板。

### 包含關鍵字

設定要通知的規則。

### 排除關鍵字

設定不要通知的規則。若某篇貼文同時命中包含與排除，最後仍會被抑制。

### `?`

顯示關鍵字規則說明與範例。

### `儲存`

儲存包含與排除關鍵字。

如果目前正在監控中，儲存後會依新規則重新掃描。

### `開始 / 暫停`

切換監控狀態。

目前語義要注意：

- 畫面顯示 `開始` 時，按下去代表重新開始目前社團監控
- 這會清掉目前社團的 seen baseline，並立即重新掃描
- 不是單純沿用原本舊狀態繼續跑

### `查看紀錄`

查看最近符合關鍵字且已通知的全域紀錄，也可以清空全部通知紀錄。

### `設定`

可調整：

- 浮動刷新 / 固定刷新
- 自動載入更多項目
- 開始後自動調整成最新排序
- 每次目標掃描項目數
- 通知通道：桌面通知、`ntfy`、Discord Webhook
- `ntfy topic`
- Discord Webhook URL

目前設定範圍是「同一社團共用」。同一個社團內的社團貼文模式與單篇貼文留言模式，會共用包含 / 排除關鍵字、通知通道、通知端點、刷新秒數、暫停狀態與目標掃描數。

### `除錯`

顯示或隱藏 debug 面板。

當 Facebook DOM 變動、貼文抽取失敗、或通知條件判斷不符合預期時，這是主要診斷入口。

debug 面板的長網址會在畫面中自動換行，避免撐破面板寬度；按「複製」時會輸出一欄一行的純文字，方便貼到 issue 或對話視窗。

debug 面板也會顯示最上方項目快篩狀態與略過原因。留言模式若沒有跳過深度掃描，可先看「快篩略過原因」判斷是排序尚未辨識、首次快取尚未建立，或最上方留言已變更。

debug 面板也會列出本輪收集策略、是否允許捲動收集，以及掃描前自動排序調整的嘗試結果。若留言模式關閉自動載入更多項目，收集策略會顯示為只使用目前已載入 DOM；開啟時則會顯示為留言多視窗保守捲動。

### 主面板拖曳

主面板標題列可拖曳，重新整理後會保留上次位置。

## 關鍵字規則

包含關鍵字與排除關鍵字都使用同樣規則：

- `;` 表示 OR
- 空格表示 AND

### 範例 1

```text
搖滾;6880;5880
```

只要出現 `搖滾` 或 `6880` 或 `5880` 就通知。

### 範例 2

```text
搖滾 6880;搖滾 5880
```

代表：

- `搖滾` 且 `6880`
- 或 `搖滾` 且 `5880`

兩者任一成立才通知。

### include / exclude 判斷原則

- include 為空時，所有貼文都先視為可通知
- exclude 只要命中任一規則，就算 include 命中也會被抑制

## 預設值

第一次安裝、且尚未有任何已儲存設定時，腳本會預設填入：

- `包含關鍵字`: `4/4 熱區; 4/4 109; 4/4 117`
- `排除關鍵字`: `徵`

預設狀態為 `已暫停`，需要手動按 `開始` 才會真正進入監控。

## 設定說明

### 啟用浮動刷新

開啟後，腳本會在你設定的最小與最大秒數之間隨機挑一個時間刷新。

例如：

- 最小刷新秒數：`25`
- 最大刷新秒數：`35`

代表每次刷新會落在 25 到 35 秒之間。

### 固定刷新秒數

當 `啟用浮動刷新` 關閉時，改用固定秒數刷新。

### 自動載入更多項目

開啟後，腳本會在單次掃描中以保守捲動方式多抓幾個視窗的內容，再做去重與比對。

在社團貼文模式中，這代表載入更多貼文；在單篇貼文留言模式中，這代表沿著留言區保守捲動並嘗試載入更多留言。掃描結束後會盡量把頁面捲動位置復原。

### 開始後自動調整成最新排序

預設開啟。開始監控後，腳本會在掃描前保守嘗試把目前頁面的排序切到此模式偏好的最新排序：

- 社團貼文模式：`新貼文`
- 單篇貼文留言模式：`由新到舊`

若 Facebook 選單尚未載入、按鈕無法辨識或選項無法點選，腳本不會硬性重試；主面板會顯示目前排序與建議狀態，掃描會回到完整保守流程。

例行掃描會先做「最上方項目快篩」。社團貼文模式會比對最上方貼文；單篇貼文留言模式會比對最上方留言。若最上方項目和上一輪相同，腳本會沿用上一輪完整掃描快取並跳過深度捲動。

### 每次最多掃描項目數

這個數字代表單次掃描的目標項目數。社團貼文模式代表貼文數，單篇貼文留言模式代表留言數。

腳本會盡量湊滿你設定的篇數，但若達到安全掃描上限仍不足，實際值可能少於目標。

- 預設：`5`
- 上限：`10`

### `ntfy topic`

需要勾選 `ntfy` 通道並填寫 `ntfy topic`，符合條件的通知才會同步送到 `ntfy`。

### Discord Webhook URL

需要勾選 Discord Webhook 通道並填寫 Discord Webhook URL，符合條件的通知才會同步送到指定 Discord 頻道。

未勾選通道或端點留空，都不會傳送該遠端通知。

### 測試通知

會依目前勾選的通道送出一則測試通知，方便確認桌面通知、`ntfy` 與 Discord Webhook 是否正常。

測試通知不會寫入貼文去重資料。

## `ntfy` 設定

如果你希望人在外面時也能收到提醒，可以設定 `ntfy`。

### 步驟

1. 在手機安裝 `ntfy` App
2. 在 App 內新增一個 topic，例如 `my-facebook-alerts`
3. 建議 topic 使用英文字母、數字、減號或底線
4. 回到 Facebook 社團頁，在腳本面板中按 `設定`
5. 勾選 `ntfy` 通道
6. 將相同 topic 填入 `ntfy topic`
7. 按一次 `測試通知` 確認手機是否收到提醒

### 常見不會送出的情況

- `ntfy topic` 是空白
- `ntfy topic` 已清空並重新儲存
- `ntfy` 通道未勾選
- 網路異常，或 `ntfy` 端暫時沒有成功接收通知

## Discord Webhook 設定

如果你希望通知直接送到 Discord 頻道，可以設定 Discord Webhook。

### 步驟

1. 在 Discord 選擇目標頻道
2. 進入 `編輯頻道`
3. 點選 `整合` → `Webhooks` → `新 Webhook`
4. 複製 Webhook URL
5. 回到 Facebook 社團頁，在腳本面板中按 `設定`
6. 勾選 Discord Webhook 通道
7. 將 URL 貼到 `Discord Webhook URL`
8. 按一次 `測試通知` 確認該頻道是否收到提醒

### 常見不會送出的情況

- `Discord Webhook URL` 是空白
- `Discord Webhook URL` 已清空並重新儲存
- Discord Webhook 通道未勾選
- Webhook 已失效、沒有權限，或 Discord 端暫時沒有成功接收通知

## 通知與去重邏輯

為了避免重複提醒，腳本會優先用貼文本身的穩定識別資訊判斷是否為同一篇文。

大致優先順序是：

1. 穩定 `postId`
2. 可還原的 canonical permalink
3. 作者、文字與其他 fallback signature

這代表：

- 能拿到穩定 `postId` 時，去重最可靠
- 某些特殊貼文型態如果拿不到完整 permalink，仍會盡量從 fallback 線索補
- 少數貼文型態仍可能因 Facebook 自身 DOM / link 結構不穩而無法完全避免誤差

### seen 與 history

- seen dedupe 會依掃描 scope 分 bucket 保存。社團貼文模式使用社團 ID；單篇貼文留言模式使用 `社團ID:post:父貼文ID:comments`
- 通知 history 則是全域清單，保留最近 10 筆

### 多社團視窗

- include / exclude 關鍵字、通知通道與端點、刷新設定與暫停狀態依目前社團 ID 保存
- dedupe / baseline 依目前掃描 scope 保存；不同單篇貼文留言頁會有各自的 seen baseline
- 你可以同時開多個不同社團視窗，各自維持自己的設定與已看過狀態
- 同一社團的多個留言視窗會共用社團設定，但留言 seen baseline 會依父貼文分開
- 不建議同時開兩個以上視窗監視同一個社團貼文 feed，避免 refresh / baseline 操作互相干擾
- 第一次升級到這個版本時，舊版全域設定會在首次讀取某個社團時搬進該社團 bucket

建議操作方式：
- 視窗 A 只開社團 A
- 視窗 B 只開社團 B
- 若要監視同一社團內不同單篇貼文留言，可以分別開在不同單篇貼文頁；設定共用，seen baseline 分開

## 已知限制

- Facebook DOM 結構會變動，selector 可能失效
- 某些貼文型態仍可能抓不到穩定 `postId`
- 某些特殊貼文或外站分享仍可能抓不到可開啟的 permalink
- `timestampText` / `timestampEpoch` 目前保留欄位形狀，但暫不做貼文時間解析
- 瀏覽器背景分頁節流會影響 refresh 與自動捲動的準時性
- 社團貼文 feed 多視窗情境目前仍以「不同社團各開一個視窗」為支援邊界
- 單篇貼文留言模式依賴 Facebook 當下 DOM 與留言捲動容器；若 Facebook 沒有載入新留言，實際掃描數可能低於目標項目數
- 最上方快篩依賴目前模式的最新排序。若自動排序關閉，或 Facebook 選單未載入 / 無法點選，腳本仍會回到完整保守掃描
- 此腳本依賴你已登入 Facebook，且社團內容能正常載入

## 建議手動檢查

做完設定後，建議至少確認：

1. 面板可正常顯示
2. `開始 / 暫停` 可切換
3. 關鍵字規則符合你的預期
4. 測試通知可正常送出
5. debug 面板能看到掃描摘要
6. 符合條件的新貼文不會被重複通知
7. 若同時開兩個不同社團視窗，兩邊的關鍵字與掃描結果不會互相覆蓋
